# 三階段 × 11 工件操作手冊（Phase Playbook）

> 給執行本 skill 的 agent：這裡是每一步「具體怎麼做」的操作說明，以及 A3 該選哪個選項的判斷準則。象限與色彩定義見 `quadrants.md`；提問排序見 `interview-heuristics.md`。

---

## 階段 A — 實施前（Pre-implementation）

這是 11 個示範中的 8 個所在的階段，因為**在寫下第一行程式碼之前，是找到未知數最便宜的地方**。

### A1. 盲點掃描 Blindspot Pass — 對付未知未知

**何時做**：幾乎每次進陌生/高風險模組都該做，是階段 A 的預設起點。

**操作步驟**：
1. 讀取目標模組/資料夾的程式碼結構（檔案數、主要職責）。
2. 跑 `git log --oneline -n 50 -- <path>` 看近期變動；用 `git log --diff-filter=R --all` 或搜尋 commit message 含 "revert" 找出被撤銷的嘗試。
3. 讀相關 migration、設定檔、測試 fixture——這些最容易藏「慣例」與「歷史教訓」。
4. 若目標範圍大（多模組、上百筆 commit），**派一個 read-only 探索 subagent** 去做上述 1–3 步並回傳結構化結論，主 agent 只組裝工件，避免 context 被檔案內容淹沒。
5. 產出 5–8 張「盲點卡」，每張標明：
   - 類型：地雷 Landmine／慣例 Convention／失落的概念 Missing concept／歷史教訓 History
   - 為什麼會咬你（Why it bites）
   - 規避作法
6. 把所有卡片濃縮成一段「改良版實作 prompt」（使用者可直接複製去下一步用）。
7. 用 `templates/blindspot-cards.html` 輸出。**此步嚴禁寫任何實作程式碼。**

**完成訊號**：使用者看過卡片後說「可以，繼續」或自己補充遺漏的地雷。

---

### A2. 訪談 The Interview — 對付已知未知

**何時做**：功能目標明確、但還有多個未拍板的決策時（幾乎每次非 trivial 功能都該做）。

**操作步驟**：
1. **主 agent 親自進行，不得外包給 subagent**——互動連續性會斷。
2. 一次只問一題，依「爆炸半徑」排序（見 `interview-heuristics.md`）：越會動搖架構的問題越先問。
3. 使用者回答後才決定下一題，通常 5–8 題，直到重大不確定性清空。
4. 收斂成決策表（決策／使用者的選擇／理由／影響範圍）＋一段可直接貼上的實作 prompt。
5. 用 `templates/interview-decisions.html` 輸出。

**完成訊號**：使用者確認決策表無誤，或明說「先這樣，其餘之後再調整」。

---

### A3. 按需選項 — 四選一（或多選）判斷準則

A3 不是必經步驟，而是依任務性質**挑選 0～多個**適用的選項。判斷方式：

| 徵狀 | 選用 | 對付象限 | 對應模板/結構 |
|---|---|---|---|
| 使用者講得出「想要什麼感覺」但講不出精確參數、術語 | **教學（詞彙階梯）** | 已知未知（缺詞彙） | 沿用 `interview-decisions.html`：把「決策表」換成「詞彙階梯表」＋ before/after 範例，最後輸出精準化 prompt |
| 是一個**開放式設計/架構決策**，沒有「正確答案」，只有取捨 | **多方向並排** | 已知未知（開放式決策） | 沿用 `blindspot-cards.html` 卡片網格：每卡是一個方向，欄位換成「適用情境／取捨／採用-捨棄 chips」 |
| 決策藏在**互動流程/介面**本身，口頭講不清楚 | **假原型（Mock）** | 已知未知（互動決策） | 以 `artifact-base.html` 骨架客製一份**純前端可點擊、不接邏輯**的原型；每個未定案的地方做成可切換選項，選擇會被記錄成清單。這一步**嚴禁**接任何真實商業邏輯或後端 |
| 目標模糊（例如「降低流失」「提升效能」），但**解法其實藏在既有程式碼裡** | **紮根發想（10 個方案）** | 未知已知（解法藏在 codebase） | 沿用 `blindspot-cards.html` 卡片網格：讀 codebase 生 10 個方案，沿「今天能上→長押注」排序，每卡附「有共鳴」核取方塊 |
| 使用者要求「照著某參考做／移植某 pattern」 | **理解驗證表** | 未知已知（誤解風險） | 沿用 `interview-decisions.html` 的表格結構：欄位換成「參考段落／對應實作／gotcha／邊界案例」，使用者確認無誤才可動手 |

多個徵狀同時成立時可疊加使用（例如先做理解驗證表，再做多方向並排）。若都不成立，可跳過 A3 直接進 A4。

### A4. 可調整的計畫 The Tweakable Plan — 對付已知未知

**何時做**：A1–A3 產出後，要進入實作前的最後一步，把散落的決策收斂成一份可執行計畫。

**操作步驟**：
1. 整合前面所有步驟的產出。
2. 計畫的排序原則是**「最可能被使用者改動的決策」在最上面**，而非依執行順序。
3. 關鍵 schema/型別決策附上 1–2 個可切換的替代方案與取捨。
4. 把純機械性、沒有決策的步驟摺疊收到最底部。
5. 沿用 `templates/interview-decisions.html` 的決策表＋可複製 prompt 樣式來呈現（欄位換成「決策／建議排序理由／替代方案／是否機械性」）。

**完成訊號**：使用者對計畫拍板，只在少數旗標決策上回饋——這代表計畫排序有效導引了他的注意力。

## 階段 B — 實施中（During）

### B1. 決策日誌 Implementation Notes — 把當場冒出的未知固定成文字

**何時做**：任何長時間、可能過夜或多輪的實作委派任務。若任務很短（幾分鐘內完成、無分歧點），可省略。

**操作步驟**：
1. **由正在實作的 agent 自己維護**，不開 subagent——只有它自己知道當下偏離了什麼。
2. 每當偏離計畫、或遇到計畫沒交代的岔路，記一筆：
   - 怎麼選的
   - 為什麼（尤其若選了保守做法）
   - 先擱置了什麼疑點
3. 做完後把日誌濃縮成「下一輪要折進計畫的 3 個要點」。
4. 用 `templates/decision-log.html` **持續更新同一份**工件（不要每筆都開新檔案）。

**完成訊號**：實作完成，3 個要點已產出並準備餵回下一輪計畫。

## 階段 C — 實施後（Post）

### C1. 說服文件 The Buy-In Doc — 對付審核者的已知未知

**何時做**：改動完成、要送 review／merge／給利害關係人看之前。

**操作步驟**：
1. 開場先做**視覺化演示**（先展示、後文字；reduced-motion 時降級為靜態分鏡）。
2. 站在審核者立場列出每一個可能的反對意見，逐條用證據（測試結果、benchmark、log）預先回應。
3. 明確標出「誰需要簽核哪一部分」。
4. 用 `templates/buy-in-doc.html` 輸出。

**完成訊號**：使用者確認異議清單涵蓋了他預期會被問到的問題。

---

### C2. 合併前測驗 Quiz Me Before I Merge — 對付「以為看懂了」的假象

**何時做**：大 diff 要送審或自我驗收時，尤其是高風險區域（付款、權限、資料遷移）。

**操作步驟**：
1. 若 diff 很大，可派 subagent 先摘要 diff 的關鍵行為，出題與連回段落的邏輯仍由主 agent 掌控。
2. 先給一張改動的心智模型圖，與 2–3 個「不明顯但重要」的行為。
3. 出 5–6 題**決策題**（不是背誦題）：都是「事故或 review 時必須判斷對」的題目。
4. 答錯的題目連回對應說明段落；全對（例如 6/6）才算通過。
5. 用 `templates/merge-quiz.html` 輸出。

**完成訊號**：使用者（或 reviewer）全對，解鎖「可以把這個改動解釋給半夜被 call 起來的人聽」的通過訊息。

## 跨階段通則

- 每一步結束都要**暫停**，把工件交給使用者確認，再問是否進入下一步；不要一口氣把 A→B→C 全部跑完。
- A 階段的掃描/訪談/A3 選項，**絕不預先寫實作程式碼**；程式碼只在使用者確認 A4 計畫之後才開始寫。
- 全程用四象限與三階段語言標註目前這一步在對付什麼，鞏固使用者的心智模型（見 `quadrants.md`）。
- 檔名慣例：`unknowns-<phase>-<step>-<slug>.html`，例如 `unknowns-A1-blindspot-pass.html`、`unknowns-A3-teach-me.html`、`unknowns-B1-decision-log.html`。
