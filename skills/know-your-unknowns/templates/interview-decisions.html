訪談決策表：clip 匯出改後端渲染

依「架構爆炸半徑」由大到小排序提問（見 references/interview-heuristics.md）；一次只問一題，答完才問下一題。以下是問答歷程與收斂後的決策表。

問答歷程

Q1 · 資料模型／系統邊界（最大半徑）渲染要同步等待還是背景 job＋輪詢？›

使用者回答

背景 job，前端輪詢一個狀態端點。

Q2 · 系統邊界signed download URL 的 TTL 要多久？過期後行為是什麼？›

使用者回答

30 分鐘，起算點是渲染完成時；過期後回傳 410，前端顯示「請重新匯出」。

Q3 · 錯誤處理舊的前端匯出路徑要保留當 fallback 還是整個移除？›

使用者回答

保留兩週當 fallback（用 feature flag 控制），確認新路徑穩定後再移除。

Q4 · 資料模型標註（annotation）在伺服器端要怎麼重新投影？›

使用者回答

需要新增一個「螢幕座標 → 影格座標」的轉換函式，並在匯出時記錄使用的影片解析度以確保比例正確。

Q5 · API 形狀渲染失敗時，狀態端點要回什麼？前端怎麼呈現？›

使用者回答

回傳分類過的錯誤碼（暫時性/永久性）；暫時性顯示「重試」按鈕，永久性顯示客服聯絡窗口。

決策表

依爆炸半徑由大到小排序；每列可直接當作實作合約引用。
決策使用者的選擇理由影響範圍
渲染執行模式背景 job ＋ 前端輪詢避免長時間佔用 request 連線，並與現有 queue 慣例一致需新增狀態端點與輪詢邏輯；影響前端匯出流程 UI
Signed URL TTL30 分鐘，起算點＝渲染完成時避免使用者拿到「還沒生成完」的連結就過期影響下載端點與過期錯誤處理（410）
舊路徑去留保留兩週 fallback，用 feature flag 控制降低一次性切換的風險，可隨時回退需要維護兩套路徑兩週，增加暫時性複雜度
標註座標轉換新增「螢幕座標→影格座標」轉換函式兩者座標系不同，是目前程式碼裡沒有文件化的隱藏概念（見 A1 盲點卡 04）影響匯出管線的標註疊圖步驟
失敗訊息分類暫時性／永久性兩類，各自對應不同 UI符合團隊 i18n/errors.ts 白名單慣例（見 A1 盲點卡 06）影響狀態端點回傳格式與前端錯誤處理元件

可直接拿去實作的 prompt

實作「clip 匯出改伺服器端 FFmpeg 背景渲染」，請遵守以下已拍板的決策： (1) 渲染走背景 job，前端輪詢狀態端點，不要同步等待； (2) signed download URL TTL 為 30 分鐘，起算點是「渲染完成時」，過期回傳 410 並提示「請重新匯出」； (3) 舊的前端 MediaRecorder 路徑保留兩週當 fallback，用 feature flag 控制，兩週後再評估移除； (4) 新增「螢幕座標 → 影格座標」轉換函式處理標註疊圖，記錄匯出時使用的影片解析度； (5) 狀態端點的錯誤要分類為暫時性/永久性，前端依分類顯示「重試」或「聯絡客服」。 以上決策已與使用者確認，可以開始實作。
