心智模型

改動前瀏覽器端 MediaRecorder 即時渲染
→
改動後背景 Job ＋ FFmpeg Worker 非同步渲染
→
交付方式Signed URL（30 分鐘 TTL）

核心變化：渲染從「使用者裝置上、同步、即時」變成「伺服器上、非同步、輪詢查詢結果」。這改變了失敗模式、時間感、與安全邊界。

不明顯但重要的行為

Signed URL 的 TTL 起算點改成「渲染完成時」，不是「請求發起時」

這代表使用者從按下匯出到連結真正過期，實際可用時間可能遠超過 30 分鐘（等 worker 排隊 + 渲染的時間都不算在內）。

舊的瀏覽器端路徑用 feature flag 保留兩週，不是立刻刪除

這兩週內，同一個匯出按鈕在 flag 開/關時走的是完全不同的程式碼路徑，debug 時要先確認使用者當下走的是哪一條。

渲染 worker 是獨立資源池，併發數上限只有 2

尖峰時段第 3 個以後的匯出請求會排隊等待，不是立刻並行處理——這不是 bug，是刻意的資源隔離設計（見決策日誌）。

測驗（6 題，需全對才算通過）

Q1. 使用者按下匯出後，若此時後端 worker 佇列已滿，使用者體感上會發生什麼？
正確答案是「立刻收到已排入佇列的回應」——渲染是背景 job，前端靠輪詢查詢狀態。詳見 「渲染 worker 是獨立資源池」。
Q2. 使用者按下匯出 5 分鐘後，worker 才排到並完成渲染。此時 signed URL 還剩多少可用時間？
正確答案是「完整 30 分鐘」——TTL 起算點是渲染完成時，不是請求發起時。詳見 「Signed URL 的 TTL 起算點」。
Q3. Feature flag 關閉（走舊路徑）時，這次改動新增的後端 worker 程式碼會被執行嗎？
正確答案是「不會」——兩條路徑是完全獨立的程式碼路徑。詳見 「舊的瀏覽器端路徑用 feature flag 保留」。
Q4. 渲染 worker 因記憶體不足而 crash，使用者會看到什麼？
正確答案是「依既有 queue 重試慣例」——這是團隊既有慣例（見 A1 盲點卡 03），不是重新發明一套錯誤處理。詳見 「渲染 worker 是獨立資源池」。
Q5. 標註（annotation）在伺服器端渲染時，座標是怎麼對齊到影格上的？
正確答案是「透過新增的轉換函式」——這是這次改動裡最容易被忽略的隱藏概念（見盲點掃描與決策日誌）。
Q6. 兩週後移除 feature flag 與舊路徑時，最應該先確認什麼？
正確答案是「確認呼叫量與錯誤率」——這是說服文件與決策表裡明訂的下線前提條件，不是單純的時間到期。
