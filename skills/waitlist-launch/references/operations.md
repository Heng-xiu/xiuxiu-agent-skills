# Waitlist 上線 — 驗收清單與上線後營運(PM 章節)

來源:fare-waitlist(early-fare.hengshiou.com)一次成功上線的實戰萃取。本章節只談「上不上線該由誰決定、上線後看什麼、什麼時候動什麼」,技術實作細節與文案模板見其他章節。

---

## 1. 上線 gate 檢查清單

分兩級。**必須項任一沒過,不准上線**——這不是品味問題,是決策邊界:凡涉隱私/防濫用/度量的項目,PM 沒看到證據不能簽字放行。

### 必須(不過不上線)

- [ ] **隱私 consent 文案已就位且誠實**:明確寫「只通知上線這件事、不做他用、不轉第三方」,且程式行為與文案一致(不能文案說不轉三方,結果 Resend/其他服務真的轉了)。
- [ ] **退訂機制存在且可用**:即使是 MVP,也要有一鍵退訂或明確的退訂聯絡方式,不能只承諾不落地。
- [ ] **不轉第三方的邊界有技術對應**:第三方同步(如 Resend)用途單一化(僅供 email 發送),沒有把名單導向廣告/再行銷用途。
- [ ] **防濫用三層都就位並實測**:蜜罐 + 平台原生限流 + 人機驗證(如 Turnstile),且做過三個契約案例(正常/重複/蜜罐)驗證,不是「寫了但沒測過」。
- [ ] **smoke test 全綠**:本機端到端 + 線上關鍵端點(健康檢查、分享卡圖片、meta 標籤)都跑過且有證據(截圖或 curl 輸出),不是「應該沒問題」。
- [ ] **analytics 已就位**:沒有可用的流量/轉換追蹤前,不開始對外導流——沒有度量的導流等於盲飛,花的是不可回收的初次曝光。
- [ ] **OG 分享卡正常**:社群分享的預覽圖與標題正確渲染(親眼看過渲染結果,不只是「程式跑完了」),因為分享卡是第一波流量的入口,壞了會直接損失轉換且難以補救(平台快取)。

### 應該(建議做,沒做不擋上線,但要記錄成 backlog)

- [ ] personal footer / 導流回作者其他管道
- [ ] referral 分享迴圈(推薦碼、社群轉發按鈕)
- [ ] 社會證明人數顯示
- [ ] 名單保溫 email 草稿已備妥(不必發,但知道要發什麼)

## 2. 成功指標框架

Waitlist 階段的目的是「驗證需求 + 累積種子名單」,不是「衝大數字」。該看的指標都要能回答「這個管道值不值得繼續投入」。

### 該看的指標

- **各渠道報名轉換率(依 ref 歸因)**:每個導流渠道(主站、Substack、社群貼文、社群心得文…)各自帶不同的 `ref` 參數,分開算「該渠道曝光 → 報名成功」的轉換率。沒有歸因就無法決定下一步把時間花在哪個渠道。
- **referral 係數**:`source` 以 `r_` 開頭(代表由既有名單成員的推薦連結帶來)的報名數佔總報名數的比例。這個數字回答「產品/文案本身有沒有讓人願意主動轉發」,是判斷要不要投資 referral 機制的核心依據。
- **名單成長曲線(而非單點總數)**:看的是斜率變化——哪個事件(發文、上首頁、寄信)讓曲線明顯拐點,用來反推哪個動作真的有效。

### 不要看的虛榮指標

- **單日/單次的曝光數或點擊數**(未落地成報名的曝光,對 waitlist 階段沒有決策意義)。
- **社群貼文的讚數/愛心數**:與轉換率脫鉤,容易誤導成「內容很紅但沒人報名」還誤判成功。
- **累積總人數本身當作唯一 KPI**:總數會掩蓋渠道品質差異——1000 人裡 900 人是同一篇爆紅但完全不相關的貼文帶來的路人,價值遠低於 100 人裡 60 人是精準渠道帶來的。

## 3. 上線後營運節奏

- **名單保溫**:名單破百(或達到有意義的樣本量)後,發 2-3 封教育型 email(如 Resend broadcast)。目的不是催促上線,是把「單純等工具的人」轉成「認同作者觀點、願意持續關注的人」——內容應該是方法論/案例,不是「還沒上線敬請期待」這種空信,空信只會增加退訂率。
- **社會證明人數顯示門檻**:只有名單數超過門檻(如 100)才顯示「已有 N 人排隊」。低於門檻不顯示的原因是:小數字(如「已有 12 人」)傳達的訊號是「這東西還沒被驗證」,反而拉低信任度,不如不顯示、留給痛點文案自己說服人。
- **何時該檢討文案**:設一個轉換率基準(例如某渠道報名轉換率低於同類渠道均值,或連續兩個渠道都低於預期),觸發後才回頭改標題/痛點句/社會證明卡,而不是憑感覺隨時改——改太頻繁會讓歸因失真(改動前後的樣本不可比)。

## 4. 風險與決策邊界

**要回頭問使用者(PM/orchestrator 不能自行拍板)的動作:**

- 任何花錢的決策(付費推廣、升級第三方服務方案、購買網域等)。
- 任何外發的內容(對外發布的貼文、email broadcast 內容、公開頁面文案變更影響 consent 承諾的部分)。
- 對既有名單發 email(即使是「教育型」也要先過目內容,因為這是對已建立信任的使用者發聲,出錯的代價是退訂/信任流失,不是能重來的)。
- 任何觸碰隱私邊界的變更(例如新增第三方同步、擴大資料用途)。

**可自動做的動作:**

- 技術實作(依既定架構決策落地)、本機/線上 smoke test、程式碼層面的防濫用強化、依既有文案框架做的小幅措辭調整(不涉及 consent 承諾變動)。

**deploy 權限的教訓**:多任務改動同一份核心檔案(如 index.html)時,依序派工而非並行,避免 Edit 衝突;每個 subagent 任務都要有明確邊界(只准改哪些檔)、Done 定義、要求附實測證據,**不准 commit/deploy**——deploy 權限收在 orchestrator 手上,逐任務驗收 diff 與關鍵產物(圖要親眼看過,不能只信 subagent 回報「做完了」),最後統一一次 deploy。這條紀律的價值在於:任何一個 subagent 的錯誤在合併前就能被攔下,而不是上線後才發現。

## 5. 迭代 backlog 模板

這次沒做但下次可考慮的項目,每項附「什麼訊號出現才值得做」——沒有訊號前不要因為「順手」就做,會分散對核心轉換率的關注。

| 項目 | 觸發訊號 |
|---|---|
| A/B 標題測試 | 單一渠道流量已穩定到能在合理時間內跑出統計意義的差異(例如單渠道週報名數已達兩位數以上),且轉換率長期低於預期基準 |
| Waitlist 排名/遊戲化(如「你排在第 N 位,分享可提前」) | referral 係數已驗證偏低(現有分享機制帶來的轉換不明顯),需要額外誘因拉高分享意願時才加,不要在還沒驗證基礎 referral 迴圈有效前就疊加複雜度 |
| 多語系 | 已有明確跨語系流量訊號(例如 analytics 顯示非中文語系訪客佔比顯著,或有目標渠道是外語社群),不要為了「將來可能需要」預先做 |
| 更豐富的社會證明(如具名見證、案例牆) | 名單成長到有真實使用者反饋可引用時,不要用假數據或空泛敘述硬撐 |
| 名單分群(依渠道/意圖標籤做差異化保溫內容) | 名單規模與渠道多樣性都到一定量(單一保溫內容已無法對不同渠道來源的人都有效)時才值得做分群成本 |
