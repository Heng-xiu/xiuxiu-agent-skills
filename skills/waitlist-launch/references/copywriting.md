## 1. 頁面文案公式

### 1.1 H1 痛點句式

**公式**:`[具體情境句:誰 + 行為頻率/場景],` + `<span class="hl">[痛點結論句,你已經在承受的損失]</span>`

- 情境句要讓讀者「對號入座」——具體到頻率或場景,不要泛泛而談。
- 結論句才是 highlight 對象,且**只 highlight 結論句、不整句上色**——情境句是鋪陳,結論句才是鉤子,顏色要落在讀者心裡「咦」的那一下。

**fare 實例**:
```html
<h1>
  一年飛兩趟同一條航線,<br />
  <span class="hl">你的來回票很可能買貴了。</span>
</h1>
```
「一年飛兩趟同一條航線」= 情境句(誰:常飛同航線的人;頻率:一年兩趟);「你的來回票很可能買貴了」= 結論句,上品牌綠(`--accent`)。

**換題目怎麼填**:先問「這個產品鎖定的人,具體會用什麼頻率/場景遇到這個問題?」把答案寫成情境句;再問「他們現在正在付出什麼代價但不自知?」寫成結論句、上色。避免結論句寫成功能描述(如「我們幫你省錢」),要寫成**讀者現況的診斷**(如「你可能買貴了」)。

### 1.2 Lede 結構

**公式**:機制一句(怎麼做到的,不用術語) + 誠實勸退(什麼情況下我們會老實說不值得) + CTA(留什麼資訊 + 給什麼時間承諾)。

**fare 實例**(逐句拆解):
> 把兩趟行程的機票「交錯開」,有時比各買一張來回票便宜好幾千。〔機制一句〕
> 我們一次幫你算完所有開法,連延誤、換機場、多住一晚的風險都折算成錢——真的沒省頭,也會老實跟你說。〔機制延伸 + 誠實勸退〕
> 工具正在最後打磨,留個 email,**上線第一時間通知你**。〔CTA:留 email + 時間承諾〕

**換題目怎麼填**:機制一句要能讓外行人一次看懂「產品做了什麼動作」;誠實勸退**必須是真的**(這句話的可信度全靠它是不是言之有物,不能空喊「我們很誠實」);CTA 要給明確時間點(「上線第一時間」比「盡快」有效,因為它是可驗證的承諾)。

### 1.3 三個賣點 bullet

**公式**:checkmark icon + 動詞開頭,講清楚系統實際做什麼;第三條保留給「誠實/篩選」型賣點,呼應反推銷語氣。

**fare 實例**:
1. 自動比較一般來回、拆單程、交錯開票,依風險調整後成本排名〔涵蓋範圍〕
2. 把延誤、換機場、多住一晚的風險都換算成錢一起比〔差異化能力〕
3. 省太少或風險太高,直接勸你別折騰〔誠實把關,呼應反推銷〕

**換題目怎麼填**:前兩條回答「這個工具比你自己算/比競品多做了什麼」,第三條刻意放一個「工具會勸你不要用」的情境——這條不是弱化賣點,是建立可信度的必要成分,不能省略也不能移到第一條(順序上要先建立能力,再談克制)。

### 1.4 Consent 一句話公式

**公式**:「我們只會用這個 `[資料類型]` 通知你「`[唯一用途]`」這一件事,不做其他用途、不轉給第三方,`[退出機制]`。」

**fare 實例**:
> 我們只會用這個 email 通知你「工具上線」這一件事,不做其他用途、不轉給第三方,信中隨時可一鍵退訂。

四個必要元素缺一不可:唯一用途宣告、明確排除其他用途、明確排除轉三方、退出權具體化(不要寫「可隨時取消訂閱」這種空話,要寫「信中一鍵」這種操作細節)。

**換題目怎麼填**:唯一用途要跟 CTA 承諾的事一致(不能 CTA 說「上線通知」,consent 卻寫「不定期分享」),否則會自相矛盾拆自己的台。

### 1.5 社會證明卡數字對比結構

**公式**:場景 label + 具體情境 pill(地點/時間) + 舊值(劃線,弱化色) + 新值(強調色,大字) + 省下金額 pill(強調色底) + 可信度 footnote(驗證方式 + 免責提醒)。

**fare 實例**:
```
試算範例                    台北 ⇄ 東京成田 · 7月 + 8月
一般買法 · 兩張來回票        NT$ 23,574 (劃線)
交錯開票 · 我們找到的最佳解   NT$ 19,337    [省 4,237]
已通過航段順序、時間連續與轉乘可行性驗證。實際票價以查詢當下為準。
```

**換題目怎麼填**:情境 pill 一定要具體到「城市/日期」這種可被讀者反查的細節,越具體越可信;footnote 兩件事都要有——「我們怎麼驗證過」(建立可信度)+「這數字會變」(誠實免責,呼應整站的反推銷語氣,不要只挑一個寫)。

## 2. 信任語氣準則

### 為什麼「反推銷」有效

fare 頁面兩句關鍵:「真的沒省頭,也會老實跟你說」「省太少或風險太高,直接勸你別折騰」。

Landing page 讀者的預設心理防備是「這頁一定會誇大」。反推銷句直接**承認工具會勸退**,等於主動戳破讀者的防備假設——一旦讀者相信「這頁連壞消息都敢講」,前面講的好處反而更可信(可信度是靠「敢說不」換來的,不是靠「講很多好處」換來的)。這也降低留信箱的心理成本:讀者知道自己不會被推銷一個對自己沒用的東西,才願意留資料。

### 禁用詞/禁用手法

- **浮誇絕對詞**:絕對、保證、最強、革命性、顛覆、獨家秘技——這類詞會立刻觸發讀者的防備心理,跟反推銷語氣直接衝突。
- **無實據的社會證明**:「許多人已經在用」「大家都說有效」——沒有具體數字或可驗證來源的集體背書,讀者會直覺懷疑。
- **假急迫感**:「僅剩 X 名額」「限時優惠」——若不是真的,一旦被拆穿,會把整頁建立起來的誠實形象一次燒光。
- **行話黑話**:術語堆疊會讓「機制一句」失去「外行人一次看懂」的效果,違背 1.2 的公式精神。

### 用詞替代原則

- 具體數字 > 形容詞(「省 4,237」比「大幅省錢」可信)。
- 「老實說 / 直接跟你說」> 「我們保證」(承諾行為,不承諾結果)。
- 「可能」「通常」這類誠實的不確定詞,比誇大的確定詞更值得信任——前提是產品本身真的有不確定性,不要為了裝誠實而假裝謙虛。

## 3. 分享 / referral 文案

### 成功態邀請語公式

**公式**:「也有 `[同溫層描述]`?分享這個連結,`[集體利益/共同等待理由]`:」

**fare 實例**:
> 也有朋友常飛同一條航線?分享這個連結,上線時一起收到通知:

同溫層描述要精準複用 H1 的情境句(「常飛同一條航線」),讓讀者一眼認出「這是說我朋友」;集體利益要落在「一起等、一起知道」而非「幫我推廣」——分享的理由是利他,不是幫產品打廣告。

### Threads / X 預填文案公式

**公式**:`痛點一句(呼應 H1,但精簡到一句)` + `個人立場(我已經在候補名單上/我已經在用)` + `連結(不加修飾直接給)`。

**fare 實例**:
```
一年飛兩趟同一條航線,來回票很可能買貴了。這個工具幫你算「交錯開票」能省多少,
我已經在候補名單上,上線一起收到通知:
<連結>
```

三段要素缺一不可:痛點一句建立情境共鳴、個人立場證明「這是真人推薦不是機器人洗版」、連結直給不加多餘 CTA(社群平台上過度推銷的語氣會被視為廣告帳號)。實作上用平台官方 intent(如 Threads 的 `https://www.threads.com/intent/post?text=<encoded>`)預填文案 + 個人 refCode 連結,讀者按下轉發時文案已經寫好,降低分享摩擦力。

### OG 圖文字層級

在 1200×630 的畫布上,文字要分三層,字級與弱化程度分開設計:

1. **標題兩行**:呼應 H1 但精簡到能在兩行內放完(約 22–30 字),避免孤兒行(遇到「了。」單獨一行的情況要降字級或加 `nowrap` 控制斷行)。
2. **行動小字(單行)**:呼應 CTA,例如「留 email,搶先收到通知」,字級不低於 18px,確保縮圖(如聊天預覽小圖)仍可讀。
3. **網址**:用 mono 字體、弱化色(如 zinc-400),但保留可讀性——網址本身是信任錨點,不能弱化到看不見。

## 4. 曝光渠道 Playbook

渠道優先序(依 brief 既有排序展開為可執行動作,每渠道建議 ref 碼歸因):

### 1. 主站互導 — `ref=site_main`
**動作**:在主產品站(或個人網站)的顯眼位置(banner / footer / 相關文章結尾)放候補名單連結,文案沿用「情境句 + CTA」公式(參考 1.1),不需要重寫一套新文案。

### 2. SEO 長文(Substack / 部落格)— `ref=blog_<文章 slug>`
**題目公式**:`[出發地]飛[目的地][頻率情境],[方法]比[對照組]省多少?實測[N 條航線/案例]`

**範例**:「台北飛東京一年兩趟,交錯開票比兩張來回票省多少?實測 3 條航線」

長文本體用實際試算案例(呼應社會證明卡的數字對比結構),文末置入候補名單 CTA,不要在文中段落硬插推廣。

### 3. Threads / X 案例圖系列貼文 — `ref=social_thread_<日期或案例編號>`
**量產方法**:複用 og.html 同一套卡片 CSS(`.card` / `.price-old` / `.price-new` 樣式),每篇只換城市、月份、價格數字,批次截圖產出系列圖。每則貼文用不同案例(不同航線/月份),文案沿用 3. 的預填公式精簡版。ref 碼帶日期或案例編號,方便事後比對哪張案例圖轉換最好。

### 4. PTT / Dcard 軟性發文 — `ref=ptt_<板名>` / `ref=dcard_<板名>`
**姿勢**:以「心得分享 / 試算過程紀錄」發文,先給有用的分析內容(例如某條航線的實際試算過程與結果),工具連結放在文末或留言區,不置頂催促、不用重複帳號洗版。

**禁忌**:不直接開版打廣告、不用小帳操作蓋樓製造熱度、不誇大試算成效、發文前務必先讀該板的行銷/業配相關板規。

### 5. Newsletter 互推 — `ref=newsletter_<對方電子報名稱>`
**動作**:跟同溫層(旅遊/理財/生活提效類)電子報主理人談互推,對方在自己的 newsletter 提到候補名單,自己也在自己的 Substack 回推對方——雙方都用各自帶 ref 碼的連結,方便各自歸因轉換。

## 5. 上線前檢查

### OG debugger 刷快取

每次改動 og.html、meta 標籤、或重新部署後,以下平台都要重新抓取一次(社群平台會快取舊的 OG 圖,不會自動更新):

- Facebook Sharing Debugger(順便檢查 og:title / og:description / og:image 是否正確抓到)
- Twitter/X Card Validator
- LinkedIn Post Inspector
- Threads 目前沒有官方 debugger,用實際分享(或私訊自己)手動驗證預覽卡是否正確

### 社群 handle 驗證

**不能瞎猜**——曾經發生 Threads 帳號猜錯、被使用者糾正的情況。上線前檢查清單:

- 每個 social icon 連結的 handle,一律從「使用者既有網站的原始碼 grep 出來」或「直接跟使用者確認」取得,不憑印象或猜測拼字。
- footer 的每一條社群連結逐一點開實測(而不是只看 href 字串),確認導向的帳號確實是本人。
